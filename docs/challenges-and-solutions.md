# Технические решения

## Snapshot pattern для финансовых данных

**Задача:** Гарантировать неизменяемость финансовых данных в заказах даже при изменении справочников (клиенты, услуги, цены).

**Решение:** При сохранении заказа все связанные данные замораживаются в JSON-полях (clientSnapshot, vehicleSnapshot, paymentSnapshot, totalsSnapshot, salaryRuleSnapshot). Исторические отчёты и печати всегда используют snapshot-данные, а не текущие справочники.

## Branch-aware access control

**Задача:** Сотрудники должны видеть только данные своих филиалов и рабочих зон. Multi-tenant архитектура без утечки данных между филиалами.

**Решение:** Каждая сессия сотрудника содержит `currentBranchId`. Репозитории фильтруют все запросы по текущему филиалу. Связь сотрудник-филиал хранится в отдельной таблице с флагами `canOperate` и `canSwitchInto`.

## Timezone handling

**Задача:** Филиалы работают в одном часовом поясе, но система должна корректно обрабатывать даты и времена операций.

**Решение:** Операционное время зафиксировано в константе timezone. Все форматирование и парсинг дат используют эту timezone через `Intl.DateTimeFormat`.

## Repository pattern

**Задача:** Разделить бизнес-логику от доступа к данным для тестируемости и масштабируемости.

**Решение:** Каждый домен имеет read/write репозитории. Репозитории инкапсулируют Prisma-запросы и применяют branch-aware фильтрацию. Бизнес-логика использует репозитории через интерфейсы.

## Строгая типизация

**Задача:** Минимизировать runtime ошибки при работе со сложной доменной моделью.

**Решение:** TypeScript на всех уровнях — от Prisma-generated типов до UI компонентов. Domain types для бизнес-логики, Prisma types для database access.

## Role-based permissions

**Задача:** Гибкое управление правами доступа без hardcoding проверок в каждом компоненте.

**Решение:** Роль → Permission маппинг. Маршруты защищены через required permissions. UI скрывает/показывает элементы на основе разрешений пользователя.
